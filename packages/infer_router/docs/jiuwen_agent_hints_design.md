# Jiuwen Agent Hints 功能设计方案

## 1. 设计概述

openJiuwen-router是一个推理引擎的路由服务，负责接收客户端请求，根据请求元数据进行路由决策，将请求转发到不同的推理节点。可以对接不同的推理引擎，如vLLm，SGLang等。
Jiuwen Agent Hints 是路由服务中用于工作负载感知推理优化的核心机制，旨在通过agent客户端提供的请求元数据.

### 1.1 设计目标
Jiuwen Agent Hints 是系统中用于工作负载感知推理优化的核心机制，旨在通过agent客户端提供的请求元数据，实现：
- **优先级调度**：根据请求重要性调整处理顺序
- **KV缓存优化**：利用预期输出序列长度优化缓存管理
- **推测性预填充**：预热KV缓存以提高后续请求性能
- **负载均衡**：智能路由请求到最佳推理节点

### 1.2 适用场景
- 多租户环境下的优先级资源分配
- 批量请求处理中的QoS保证
- 对话式AI场景中的上下文管理
- 高并发推理服务中的性能优化

### 2.2 核心组件

1. **API服务层**：接收客户端请求，解析`jiuwenext.agent_hints`字段
2. **预处理层**：将agent_hints转换为内部路由提示
3. **路由决策层**：
   - **KV索引器**：维护全局KV缓存块视图，构建前缀树索引
   - **工作器选择器**：基于KV缓存重叠度和负载状况选择最佳工作器

5. **KV缓存系统**：使用预期输出长度优化缓存块管理和衰减策略

## 3. 数据模型

### 3.1 客户端请求
"jiuwenext": {
       "agent_hints": {
         "priority": 5,
         "osl": 256,
         "next_turn_prefill": true
       }
}
- **`priority`** controls scheduling across both the router and engine. Higher values mean "more important" at the API level; Infer router translates that into router queue ordering and backend-specific engine priority.
- **`osl`** (output sequence length) is the harness's estimate of how many tokens this request will generate. The router uses this to gauge how long a worker will be occupied, which improves load balancing. A harness can learn this over time by tracking average output lengths per tool call type.
- **`next_turn_prefill`** signals the orchestrator to begin caching this request's prefix on a likely worker before the full request is ready. This is useful when the harness knows a tool call is about to return and wants to warm the cache ahead of time.

jiuwenext字段放置在Restful请求体的最外层，作为一级field

### 3.2 详细字段说明

#### 3.2.1 `agent_hints` 字段

| 字段名 | 类型 | 默认值 | 作用 | 赋值方式 |
|-------|------|-------|------|--------|
| `priority` | 整数 | 动态计算 | 请求优先级，用于调度系统排序请求 | 由客户端根据延迟敏感度动态计算：`priority = max_sensitivity - latency_sensitivity` |
| `osl` | 整数 | 512 | 预期输出序列长度，用于KV缓存优化 | 通过配置参数 `jiuwenext_prefix_osl` 设置，支持LOW(128)/MEDIUM(512)/HIGH(2048)或自定义整数 |
| `prefix_id` | 字符串 | 自动生成 | 前缀ID，用于标识同一对话或工作流的请求 | 由 `PrefixContext` 自动生成：`{workflow_run_id}-d{depth}` |
| `total_requests` | 整数 | 10 | 预期请求总数，用于预测工作负载和优化KV缓存 | 通过配置参数 `jiuwenext_prefix_total_requests` 设置，范围1-50 |
| `iat` | 整数 | 250 | 预期请求间隔时间（Inter-Arrival Time），单位毫秒 | 通过配置参数 `jiuwenext_prefix_iat` 设置，支持LOW(50)/MEDIUM(250)/HIGH(750)或自定义整数 |

#### 3.2.2 `cache_control` 字段

| 字段名 | 类型 | 默认值 | 作用 | 赋值方式 |
|-------|------|-------|------|--------|
| `type` | 字符串 | "ephemeral" | 缓存固定策略类型 | 通过配置参数 `jiuwenext_cache_pin_type` 设置，支持： |
|       |      |       |      | - "ephemeral"：临时固定，适用于短期会话 |
|       |      |       |      | - "persistent"：持久固定，适用于长期会话 |
|       |      |       |      | - "none"：不固定，使用默认缓存策略 |
| `ttl` | 字符串 | 动态计算 | 缓存生存时间 | 由系统根据 `total_requests` 和 `iat` 动态计算： |
|       |      |       |      | - 格式："[数字]s"（秒）或 "[数字]m"（分钟） |
|       |      |       |      | - 示例：`ttl = max(1, ceil(total_requests * iat / 1000))` |

### 3.3 字段使用场景

| 字段 | 主要使用场景 | 系统组件 |
|------|------------|---------|
| `priority` | 请求调度排序 | 调度系统 |
| `osl` | KV缓存分配、衰减策略 | KV缓存系统 |
| `prefix_id` | 请求关联、工作负载分析 | 路由决策层 |
| `total_requests` | 工作负载预测、缓存TTL计算 | 路由决策层、KV缓存系统 |
| `iat` | 工作负载预测、缓存TTL计算 | 路由决策层、KV缓存系统 |
| `cache_control.type` | 缓存保留策略 | KV缓存系统 |
| `cache_control.ttl` | 缓存生存时间 | KV缓存系统 |

### 4.1 请求处理流程

1. **客户端请求发送**

       }

   - 功能: 将AgentHints转换为路由决策提示信息

4. **路由决策**
   - 功能: 使用路由提示信息寻找最佳推理服务节点

5. **调度处理**
   - 功能: 根据优先级策略调度请求

### 4.2 优先级调度实现

#### FCFS (First-Come-First-Served) 策略

#### LCFS (Last-Come-First-Served) 策略

#### WSPT (Weighted Shortest Processing Time) 策略

**最大缓存重叠度来源**：
- 通过`KV索引器`（`find_matches`方法）为输入令牌序列查找所有工作器的KV缓存匹配情况
- 结果存储在`request.overlaps.scores`中，键为工作器ID，值为匹配的块数量
- `max_overlap`是所有工作器匹配块数量的最大值

**优先级键使用**：
- 优先级键用于调度队列的排序
- 在WSPT策略中，键值越高，请求优先级越高
- 键值计算考虑了请求优先级和KV缓存重叠度：
  - 更高的`priority_jump`会提高权重，增加优先级
  - 更高的缓存重叠度会减少`new_tokens`，增加优先级
- 实现了"加权最短处理时间"策略，平衡了请求优先级和处理效率

### 4.3 KV缓存优化

#### 4.3.1 KV索引器实现

KV索引器是路由决策的核心组件，负责维护全局KV缓存块的视图：

1. **数据结构**：使用前缀树(Radix Tree)存储KV缓存块信息
2. **核心功能**：
   - 接收和处理KV事件（块存储/移除/清空）
   - 为输入令牌序列查找最佳匹配的工作器
   - 支持TTL-based过期和大小-based修剪

**KV事件类型与处理**：

1. **事件类型**：
   - **Stored事件**：当工作器存储新的KV缓存块时发送
   - **Removed事件**：当工作器移除KV缓存块时发送
   - **Cleared事件**：当工作器清空所有KV缓存块时发送

2. **事件处理流程**：
   - **接收事件**：通过事件平面（ZMQ）接收来自工作器的KV事件
   - **路由事件**：根据工作器ID将事件路由到相应的处理线程
   - **更新索引**：
     - Stored事件：将块添加到radix树中，并更新工作器的块映射
     - Removed事件：从radix树中移除块，并更新工作器的块映射
     - Cleared事件：移除工作器的所有块信息
   - **维护一致性**：确保所有并发操作的一致性和线程安全

3. **KV事件实现方式**：
   - **实现一**：依赖工作器的KV事件更新
     - 触发方式：工作器主动推送
     - 优势：实时性高，准确性好
     - 劣势：依赖工作器实现，可能有延迟
   - **实现二**：infer_router自己产生事件
     - 触发方式：路由时生成
     - 优势：不依赖工作器，控制度高
     - 劣势：需要模拟工作器状态

4. **实现二的事件产生机制**：
   - **Worker Token 管理**：将工作器视为token容器，使用LRU策略管理
   - **Store事件**：请求分配给工作器时生成，token按逆序处理
   - **Removed事件**：工作器token超过容量时生成，移除最早的token
   - **Cleared事件**：工作器失联后重新连接时生成

5. **索引类型**：
   - **单线程RadixTree**：适用于低并发场景，支持TTL和修剪
   - **并发RadixTree**：默认实现，支持高并发，使用线程池处理事件

#### 4.3.2 输出块衰减因子

**参数说明**：
- **current_osl**：当前请求已生成的输出序列长度（Output Sequence Length）
  - 来源：在生成响应过程中动态跟踪，每次生成新令牌块时递增
  - 用途：衡量请求生成的进度

- **expected_output_tokens**：预期输出序列长度
  - 来源：客户端请求中的`jiuwenext.agent_hints.osl`字段
  - 转换路径：`agent_hints.osl` → `RoutingHints.expected_output_tokens` → 传递给KV路由器
  - 用途：预测请求需要的完整输出长度

**衰减因子作用**：
- 计算方式：`1.0 - (当前输出长度 / 预期输出长度)`
- 取值范围：0.0（接近完成）到1.0（刚开始）
- 使用场景：
  1. 用于KV缓存块的保留优先级计算
  2. 影响工作器负载评估和选择
  3. 控制缓存块的临时程度标记

**实际效果**：
- 接近完成的请求（current_osl接近expected_output_tokens）的缓存块对工作器负载计算的贡献较小
- 系统更倾向于将新请求分配给这些工作器，提高资源利用率
- 减少了缓存块被过早移除的可能性，同时避免了资源浪费

### 4.4 最佳工作器选择算法

#### 4.4.1 核心选择逻辑

最佳工作器选择是KV感知路由的核心，它综合考虑以下因素：

1. **KV缓存重叠度**：输入序列与各工作器KV缓存的匹配程度
2. **工作器负载**：当前工作器的计算资源和内存使用情况
3. **请求优先级**：来自agent_hints.priority的优先级设置
4. **预期输出长度**：来自agent_hints.osl的输出序列长度预期

#### 4.4.2 选择流程

```text
┌─────────────────────────────┐
│   输入令牌序列              │
└───────────────────┬─────────┘
                    ▼
┌─────────────────────────────┐
│   计算令牌块哈希            │
└───────────────────┬─────────┘
                    ▼
┌─────────────────────────────┐
│   KV索引器查找匹配块        │
│   (overlap_scores)          │
└───────────────────┬─────────┘
                    ▼
┌─────────────────────────────┐
│   获取所有工作器负载信息    │
└───────────────────┬─────────┘
                    ▼
┌─────────────────────────────┐
│   计算每个工作器的综合分数  │
│   - 重叠度分数             │
│   - 预填充块分数           │
│   - 解码块分数             │
└───────────────────┬─────────┘
                    ▼
┌─────────────────────────────┐
│   选择综合分数最高的工作器  │
└─────────────────────────────┘
```

#### 4.4.3 综合分数计算

**综合分数计算详解**：

1. **关键参数**：
   - `overlap_weight`：重叠度权重（默认值可配置）
   - `temperature`：选择随机性参数（0.0表示贪心选择）
   - `potential_prefill_block`：潜在的预填充块数量
   - `decode_block`：需要解码的块数量

2. **分数计算公式**：
   ```text
   logit = overlap_weight * potential_prefill_block + decode_block
   ```

3. **分数含义**：
   - **potential_prefill_block**：基于KV缓存重叠度的预填充效率指标
   - **decode_block**：解码阶段的工作量估计
   - **综合分数**：平衡了缓存效率和解码工作量，分数越低表示处理成本越低

4. **工作器选择机制**：
   - **贪心选择**（temperature=0.0）：选择分数最低的工作器
   - **Softmax采样**（temperature>0.0）：基于分数概率分布进行采样
   - **平局处理**：当多个工作器分数相同时，使用树大小作为次要指标

### 4.5 推测性预填充

**推测性预填充工作原理**：

1. **触发时机**：当客户端请求中设置了`agent_hints.next_turn_prefill: true`时启用

2. **实现机制**：
   - 包装原始响应流，累积助手响应文本
   - 在当前请求**完成后立即**创建异步任务发送预填充请求
   - 不会等待任何额外时间，立即执行预填充

3. **预填充请求内容**：
   - 完整的对话历史（包含新的助手响应）
   - 设置`add_generation_prompt = false`确保生成的是下一个用户请求的前缀
   - 使用`max_tokens=1`最小化资源消耗

4. **主要优势**：
   - 预热KV缓存，减少下一个请求的延迟
   - 利用当前请求的上下文信息，提高缓存命中率
   - 最小化额外资源消耗（只生成1个令牌）

5. **使用场景**：
   - 对话式AI场景，预期会有连续的用户请求
   - 需要低延迟响应的交互式应用
   - 批量处理连续的相关请求
