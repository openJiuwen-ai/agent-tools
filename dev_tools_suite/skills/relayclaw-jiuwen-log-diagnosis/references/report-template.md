# 诊断报告模板

输出报告时优先使用这个结构。用户如果说“没看懂”，使用最后的极简模板。

## 标准报告

```markdown
## 结论
问题边界在 <层级>/<模块>，这是/不是 jiuwenclaw 问题。直接原因是 <原因>，不是 <容易误解的层级> 的问题。

## 证据链
- <本地时间>，`<日志文件>`：`<关键日志原文>`  
  说明：<这条日志说明什么>。
- <本地时间>，`<日志文件>`：`<关键日志原文>`  
  说明：<这条日志说明什么>。

## 时间线
| 本地时间 | 层级 | ID | 事件 | 含义 |
|---|---|---|---|---|
| <time> | API | invocationId=... | Created invocation | 上层创建调用 |
| <time> | API | requestId=... | jiuwen request sent | 已发到 jiuwenclaw |
| <time> | jiuwenclaw | request_id=... | LLM invoke 失败 | 模型调用失败 |

## 为什么不是另一层
<解释为什么不是 jiuwenclaw / API / 前端 / 调度器。>

## 修正策略
- 立即规避：<重新登录/刷新/重试/补发等>。
- 研发修复：<异常分类、fallback、session 恢复等>。
- 测试补充：<休眠恢复、token 过期、断网恢复等>。

## 还需要补充的日志
<如果证据不足，列出最少补充项；证据足够则省略。>
```

## 极简解释

```markdown
一句话结论：
问题出在 <层级>，不是 <另一层级>。

为什么：
日志里有 `<关键错误>`，它表示 <通俗解释>。

前端为什么显示“<文案>”：
因为 <调用链在哪一环失败>，上层只返回了兜底失败文案。
```

## 研发复盘格式

```markdown
## Root Cause
<直接根因和边界>

## Trigger
<什么触发了问题，例如休眠恢复后首次定时任务>

## Impact
<用户看到什么，是否影响主任务/投递/后台记忆>

## Evidence
1. `<file>`: `<log>`
2. `<file>`: `<log>`

## Fix Proposal
1. <代码修复>
2. <错误文案>
3. <测试用例>

## Risk
<修复风险和兼容性>
```

## 测试复现格式

```markdown
## 复现条件
- 系统：Windows
- 场景：<定时任务/休眠/登录态过期>
- 配置：<模型/渠道/任务>

## 复现步骤
1. <步骤>
2. <步骤>

## 预期结果
<应该看到什么>

## 实际结果
<实际看到什么>

## 日志证据
<关键日志>

## 初步定界
<层级和模块>
```
