# Harness 架构准则

这个 skill 应按 harness 思路维护：主技能只负责稳定编排诊断流程，领域知识、案例数据、脚本和报告格式分别放在可独立演进的资源中。

## 1. 分层职责

- `SKILL.md`: 入口和决策流程。保持短、可执行、少案例堆叠。
- `references/`: 相对稳定的知识，例如架构、日志地图、指纹、报告模板、外部工具说明。
- `cases/log_case_repo.md`: 单文件已定位问题库。记录真实问题、关键日志特征、定界、排查结论和复用注意事项，团队可直接追加。
- `scripts/`: 可重复执行的采集、解析、索引或校验脚本。
- `evals/`: 评估 prompts，用于验证 skill 是否真的先定界、再分析模块。
- `evolution/`: 自动沉淀的使用经验，不直接替代正式 references 或 cases。

## 2. 诊断 harness 流程

一次诊断应像测试 harness 一样可复现：

1. 输入归一化：确认采集包、时间窗口、用户现象和期望输出。
2. 案例候选召回：先用 `cases/log_case_repo.md` 按现象、分类、案例标题和关键日志特征找相似问题，把它们作为搜索关键词和候选边界。
3. 最小证据集：再找 `MANIFEST.txt`、API/launcher、`full.log`、关联 ID，用本次日志验证或排除候选案例。
4. 边界判定：先回答“是不是 jiuwenclaw”，再回答具体模块。
5. 深入分析：只有请求进入对应层后，才使用 history/trace、源码或外部工具。
6. 案例复用：确认同一时间窗口、关联 ID 和入口/出口日志满足复用条件后，才引用历史案例结论。
7. 报告输出：结论、证据链、为什么不是另一层、修正策略、缺失日志。
8. 经验回写：确认的新问题追加到 `cases/log_case_repo.md`；通用指纹补到 `references/`；重复操作沉淀为 `scripts/`。

## 3. 可扩展性要求

- 新问题类型优先追加到 `cases/log_case_repo.md`，不要把所有案例写进 `SKILL.md`。
- 新的稳定日志指纹放入 `references/fingerprints.md`，并在相关案例中引用。
- 新的测试分类经验放入 `references/issue-taxonomy.md`，不与单个案例混在一起。
- 短期保持单文件案例库，降低团队维护成本；只有案例量明显变大且检索困难时，再考虑由脚本从 `log_case_repo.md` 生成派生索引。
- 脚本只能做确定性提取、索引、格式校验，不应替代人的边界判断。

## 4. references 更新策略

`references/` 不是远端最新代码的实时镜像，而是“已验证、可复用、跨问题有价值”的诊断知识层。版本频繁时，按下面规则维护：

### 什么时候必须更新 references

- 发布版本改变了调用链、模块边界、关键状态机或错误传播路径，例如 invocation、E2A、fallback、outbound delivery、session memory 的语义变化。
- 新版本新增、删除或重命名关键日志字段、模块名、错误码、前端文案、trace 事件。
- 同一个日志指纹在新旧版本中含义不同，可能导致旧经验误判。
- 某个案例在 2 次以上真实问题中复现，已经从单点案例上升为通用判断规则。
- 研发明确确认了某个根因、修复点或版本行为，且它会影响后续诊断。

### 什么时候不要更新 references

- 只是一次未完全确认的问题，先写入或补充 `cases/log_case_repo.md`，不要提升为通用规律。
- 只是远端默认分支的一次未发布改动，除非它已经进入用户正在测试或发布的版本。
- 只影响某个具体用户环境、路径、账号、网络、安装状态的个例。
- 证据只来自代码猜测，没有日志、测试或研发确认支撑。

### 推荐更新节奏

- 每次发布或测试基线切换后，快速检查 `architecture.md`、`fingerprints.md`、`source-checkout.md` 是否仍匹配当前发布版本。
- 每次完成一个已确认问题后，先补 `cases/log_case_repo.md`；如果发现它改变了通用判断，再同步提升到对应 `references/` 文件。
- 当诊断报告中出现“源码版本未确认一致”“日志语义需要确认”这类标记时，后续拿到对应版本源码或研发确认后，回补 `references/`。

### 更新时必须写清楚

- 适用版本或已知生效范围，例如“4.2 采集包”“2026-05 发布线”“main 截止 commit xxx”。
- 证据来源：日志样本、源码路径/commit、研发确认、历史案例编号。
- 复用边界：什么条件下能套用，什么相似现象不能套用。

## 5. 避免的反模式

- 把 `SKILL.md` 写成超长事故复盘集合，导致入口流程变慢。
- 一开始拉源码或跑 trace，跳过案例候选召回和 API/jiuwenclaw 边界判断。
- 用相似案例直接下结论，而不核对本次时间窗口和关联 ID。
- 读取远端默认分支后，把未对齐发布版本的代码当成本次线上根因证据。
- 把未发布分支的代码行为提前写进 `references/`，导致诊断已发布版本时误判。
- 把临时未证实猜测写进 `confirmed` 案例。
- 在案例里保存敏感日志原文、token、账号、完整用户内容。
